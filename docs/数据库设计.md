# FoodAdvisor 数据库设计

> 版本：V0.3（完整业务设计版）  
> 数据库：PostgreSQL 16  
> 配套存储：OpenSearch、Redis  
> 更新日期：2026-07-15

---

## 1. 文档目标

本文档统一 FoodAdvisor 全部 EPIC 的数据模型、实体关系、字段、约束、索引、状态、删除策略和实施顺序，覆盖：

- 用户注册、登录、角色与权限
- 商家、营业时间、菜品、标签和专题
- 评价、图片、追评、版本、删除、举报、商家回复和通知
- 内容安全、人工审核、敏感预警
- 情感分析、评价标签、差评归因、口碑摘要、亮点和经营建议
- AI 探店会话、需求提取、推荐排序和依据溯源
- 批量导入、OpenSearch 同步
- 模型配置、提示词版本和 AI 调用监控
- 用户行为、榜单、运营统计
- AI 测试集、评估、反馈和审计

---

## 2. 存储职责

### 2.1 PostgreSQL

保存正式业务数据，是最终数据来源。

### 2.2 OpenSearch

保存商家、菜品、评价和知识块的全文与向量索引。所有数据必须能由 PostgreSQL 重建。

### 2.3 Redis

保存限流计数、幂等锁、登录失败计数、短期会话状态和缓存。Redis 不承担正式业务唯一存储。

## 3. 统一规范

### 3.1 命名

数据库使用小写下划线：

```text
merchant_members
merchant_id
created_at
```

Java 和接口使用小驼峰：

```text
merchantId
createdAt
```

### 3.2 主键

```sql
id BIGSERIAL PRIMARY KEY
```

外部业务对象另设 `merchant_code`、`event_id`、`trace_id`、`call_id` 等业务编号。

### 3.3 时间

统一使用 `TIMESTAMPTZ`。核心表包含：

```sql
created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

### 3.4 状态

使用 `VARCHAR + CHECK`，第一阶段不使用 PostgreSQL ENUM。

### 3.5 删除

核心数据使用逻辑删除：

```sql
deleted_at TIMESTAMPTZ
```

评论、推荐、分析、举报、审核、通知和日志保留历史。

### 3.6 JSONB

JSONB 只保存结构易变且无需高频关系查询的数据。需要关联、筛选和统计的内容必须建关系表。

## 4. 统一业务规则

### 4.1 角色

```text
USER
MERCHANT
OPERATOR
ADMIN
```

未登录返回 401；无权限返回 403。后端从认证上下文取得当前用户，不信任请求中的 `userId`。

### 4.2 商家状态

平台状态：

```text
ACTIVE
DISABLED
ARCHIVED
```

经营状态：

```text
OPERATING
SUSPENDED
CLOSED_PERMANENTLY
```

当前是否开门由营业时间计算。允许评价的商家必须同时满足：

```text
platform_status = ACTIVE
operation_status = OPERATING
deleted_at IS NULL
```

### 4.3 评价规则

- 正文 10～2000 字
- 综合评分 1～5 星
- 口味、环境、服务评分可空，填写时为 1～5
- 每条评价最多 9 张图片
- 单张最大 10MB
- 支持 JPEG、PNG、WebP
- 同一用户对同一商家仅保留一条当前有效原评价
- 再次发表等价于更新当前评价并保留历史版本
- 每条原评价最多一条有效追评
- 追评必须填写消费日期
- 新增或修改后重新执行内容安全检测与评论分析
- 待审核、驳回、隐藏、删除内容不参与公开展示、推荐、标签、摘要和有效检索

### 4.4 审核默认值

新评价默认：

```text
status = PENDING
moderation_status = PENDING
```

审核通过后：

```text
status = PUBLISHED
moderation_status = APPROVED
```

### 4.5 AI 追踪

一次业务流程可以产生多次模型调用：

- `business_trace_id`：一次业务流程
- `call_id`：一次模型调用，唯一

# 6. 核心表设计

## 6.1 users

| 字段 | 类型 | 约束/说明 |
|---|---|---|
| id | BIGSERIAL | 主键 |
| username | VARCHAR(50) | 非空、唯一 |
| password_hash | VARCHAR(255) | 非空，只保存哈希 |
| nickname | VARCHAR(100) | 可空 |
| email | VARCHAR(255) | 可空 |
| phone | VARCHAR(50) | 可空 |
| role | VARCHAR(20) | USER/MERCHANT/OPERATOR/ADMIN |
| status | VARCHAR(20) | ACTIVE/DISABLED/LOCKED |
| failed_login_count | INTEGER | 默认0，非负 |
| locked_until | TIMESTAMPTZ | 锁定截止 |
| last_login_at | TIMESTAMPTZ | 最近登录 |
| password_changed_at | TIMESTAMPTZ | 密码更新时间 |
| created_at | TIMESTAMPTZ | 创建 |
| updated_at | TIMESTAMPTZ | 更新 |
| deleted_at | TIMESTAMPTZ | 逻辑删除 |

核心约束：

```sql
UNIQUE (username)
CHECK (role IN ('USER','MERCHANT','OPERATOR','ADMIN'))
CHECK (status IN ('ACTIVE','DISABLED','LOCKED'))
CHECK (failed_login_count >= 0)
```

索引：

```sql
CREATE INDEX idx_users_role_status ON users(role, status);

CREATE UNIQUE INDEX uk_users_email_active
ON users(lower(email))
WHERE email IS NOT NULL AND deleted_at IS NULL;
```

## 6.8 reviews

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGSERIAL | 主键 |
| merchant_id | BIGINT | 商家 |
| user_id | BIGINT | 系统用户，可空用于导入 |
| import_task_id | BIGINT | 导入任务，可空 |
| review_type | VARCHAR(20) | ORIGINAL/FOLLOW_UP |
| parent_review_id | BIGINT | 追评关联 |
| rating | SMALLINT | 综合评分 |
| taste_rating | SMALLINT | 口味 |
| environment_rating | SMALLINT | 环境 |
| service_rating | SMALLINT | 服务 |
| average_spend | NUMERIC(10,2) | 人均 |
| consumption_date | DATE | 消费日期 |
| content | TEXT | 10～2000字 |
| source | VARCHAR(30) | SYSTEM/IMPORT |
| external_id | VARCHAR(200) | 外部编号 |
| source_user_key | VARCHAR(200) | 外部用户标识 |
| idempotency_key | VARCHAR(100) | 幂等键 |
| current_version | INTEGER | 当前版本 |
| status | VARCHAR(20) | PENDING/PUBLISHED/HIDDEN/DELETED |
| moderation_status | VARCHAR(20) | PENDING/APPROVED/REJECTED |
| risk_level | VARCHAR(20) | LOW/MEDIUM/HIGH |
| published_at | TIMESTAMPTZ | 发布 |
| edited_at | TIMESTAMPTZ | 编辑 |
| created_at | TIMESTAMPTZ | 创建 |
| updated_at | TIMESTAMPTZ | 更新 |
| deleted_at | TIMESTAMPTZ | 删除 |

约束：

```sql
CHECK (review_type IN ('ORIGINAL','FOLLOW_UP'))

CHECK (
  (review_type = 'ORIGINAL' AND parent_review_id IS NULL)
  OR
  (review_type = 'FOLLOW_UP' AND parent_review_id IS NOT NULL)
)

CHECK (rating IS NULL OR rating BETWEEN 1 AND 5)
CHECK (taste_rating IS NULL OR taste_rating BETWEEN 1 AND 5)
CHECK (environment_rating IS NULL OR environment_rating BETWEEN 1 AND 5)
CHECK (service_rating IS NULL OR service_rating BETWEEN 1 AND 5)
CHECK (average_spend IS NULL OR average_spend >= 0)
CHECK (char_length(btrim(content)) BETWEEN 10 AND 2000)
CHECK (current_version >= 1)
```

唯一索引：

```sql
CREATE UNIQUE INDEX uk_reviews_user_merchant_original
ON reviews(user_id, merchant_id)
WHERE user_id IS NOT NULL
  AND review_type = 'ORIGINAL'
  AND status <> 'DELETED';

CREATE UNIQUE INDEX uk_reviews_parent_follow_up
ON reviews(parent_review_id)
WHERE parent_review_id IS NOT NULL
  AND review_type = 'FOLLOW_UP'
  AND status <> 'DELETED';

CREATE UNIQUE INDEX uk_reviews_user_idempotency
ON reviews(user_id, idempotency_key)
WHERE user_id IS NOT NULL
  AND idempotency_key IS NOT NULL;
```

服务层校验：

- 追评用户和商家必须与原评价一致
- 追评必须填写消费日期
- 原评价删除后追评同步逻辑删除
- 超过30天不能修改评分

```json

  "documentId": "review-101",
  "merchantId": 1,
  "merchantCode": "M000001",
  "sourceType": "REVIEW",
  "sourceId": 101,
  "sourceVersion": 2,
  "chunkIndex": 0,
  "merchantName": "川味小馆",
  "category": "餐饮",
  "cuisine": "川菜",
  "platformStatus": "ACTIVE",
```

5. 同步失败写入 `opensearch_sync_tasks`。
6. 推荐依据在 PostgreSQL 保存快照。

# 14. 旧表迁移

当前项目曾使用：

```text
restaurants
```

正式设计统一为：

```text
merchants
```

迁移步骤：

1. 检查 `restaurants` 是否有需保留数据。
2. 迁移数据到 `merchants`。
3. 后端实体、Mapper、Service、Controller 和接口统一改为 Merchant。
4. 确认无引用后删除旧表。
5. 不允许两套商家主表长期并存。

# 15. 设计原则总结

1. PostgreSQL 是最终业务数据源。
2. OpenSearch 可重建，Redis 可过期。
3. 重要唯一性由数据库约束兜底。
4. 评价、审核、推荐和分析保留版本与历史。
5. 用户身份从认证上下文取得。
6. 商家平台状态、经营状态和当前营业状态分开。
7. 公开评价必须同时满足发布和审核通过。
8. AI 输出可追踪到模型、提示词、数据版本和真实依据。
9. 推荐分数由规则或算法计算，大模型不得编造。
10. 敏感配置不得明文存储或写入日志。
11. 统计表保存聚合结果，原始事实保留在业务表与事件表。
12. 按批次实施，不一次创建全部远期表。
