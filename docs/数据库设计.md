# FoodAdvisor 数据库设计

## 1. 文档说明

本文档用于说明 FoodAdvisor 项目的数据存储划分、PostgreSQL 核心表、表关系、唯一约束和索引设计。

当前版本为 V0.1，主要用于团队前期开发。正式建表时应根据实际业务继续补充字段和索引。

---

## 2. 数据存储划分

### 2.1 PostgreSQL

保存正式业务数据：

- 用户
- 商家
- 菜品
- 评论
- 对话会话
- 对话消息
- 推荐记录
- 推荐商家明细
- 推荐反馈
- 评论分析结果

### 2.2 OpenSearch

保存检索数据：

- 商家搜索文档
- 菜品搜索文档
- 评论搜索文档
- RAG 知识块
- 文本向量
- 检索关键词

### 2.3 Redis

保存临时数据：

- 接口访问次数
- AI 请求限流计数
- 登录失败计数
- 热门商家缓存
- 会话临时状态
- 短时间重复请求标记

Redis 不作为正式业务数据的唯一存储。

## 3. 命名规范

### 3.1 表名和字段名

PostgreSQL 表名和字段名统一使用小写下划线：

```text
users
merchant_id
created_at
recommendation_feedback
```

### 3.2 主键

主键统一使用：

```text
id BIGSERIAL PRIMARY KEY
```

### 3.3 时间字段

核心表建议统一包含：

```text
created_at
updated_at
```

如需要逻辑删除，可增加：

```text
deleted
```

## 5. 用户表 users

用于保存普通用户和管理员信息。

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100),
    role VARCHAR(20) NOT NULL DEFAULT 'USER',
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT uk_users_username UNIQUE (username)
);
```

字段说明：

| 字段 | 说明 |
|---|---|
| id | 用户主键 |
| username | 登录用户名 |
| password_hash | 加密后的密码 |
| nickname | 用户昵称 |
| role | USER 或 ADMIN |
| status | ACTIVE 或 DISABLED |
| created_at | 创建时间 |
| updated_at | 更新时间 |

## 6. 商家表 merchants

```sql
CREATE TABLE merchants (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    category VARCHAR(100),
    rating NUMERIC(3, 2),
    average_price NUMERIC(10, 2),
    address VARCHAR(500),
    longitude NUMERIC(10, 6),
    latitude NUMERIC(10, 6),
    phone VARCHAR(50),
    description TEXT,
    business_hours VARCHAR(200),
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

建议索引：

```sql
CREATE INDEX idx_merchants_name ON merchants(name);
CREATE INDEX idx_merchants_category ON merchants(category);
CREATE INDEX idx_merchants_rating ON merchants(rating);
CREATE INDEX idx_merchants_average_price ON merchants(average_price);
```

## 8. 评论表 reviews

```sql
CREATE TABLE reviews (
    id BIGSERIAL PRIMARY KEY,
    merchant_id BIGINT NOT NULL,
    user_id BIGINT,
    rating NUMERIC(3, 2),
    content TEXT NOT NULL,
    source VARCHAR(50) DEFAULT 'SYSTEM',
    review_time TIMESTAMP,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_reviews_merchant
        FOREIGN KEY (merchant_id) REFERENCES merchants(id),

    CONSTRAINT fk_reviews_user
        FOREIGN KEY (user_id) REFERENCES users(id)
);
```

建议索引：

```sql
CREATE INDEX idx_reviews_merchant_id ON reviews(merchant_id);
CREATE INDEX idx_reviews_user_id ON reviews(user_id);
CREATE INDEX idx_reviews_rating ON reviews(rating);
CREATE INDEX idx_reviews_review_time ON reviews(review_time);
```

说明：

- 外部导入评论没有系统用户时，`user_id` 可以为空。
- `source` 用于记录评论来源。
- 评论原文保存在 PostgreSQL，并同步到 OpenSearch 用于检索。

## 12. 推荐记录表 recommendations

一次 AI 推荐请求对应一条推荐记录。

```sql
CREATE TABLE recommendations (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    session_id BIGINT NOT NULL,
    user_message_id BIGINT,
    assistant_message_id BIGINT,
    query_text TEXT NOT NULL,
    parsed_constraints JSONB,
    reply_text TEXT,
    model_name VARCHAR(100),
    status VARCHAR(20) NOT NULL DEFAULT 'SUCCESS',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_recommendations_user
        FOREIGN KEY (user_id) REFERENCES users(id),

    CONSTRAINT fk_recommendations_session
        FOREIGN KEY (session_id) REFERENCES chat_sessions(id),

    CONSTRAINT fk_recommendations_user_message
        FOREIGN KEY (user_message_id) REFERENCES chat_messages(id),

    CONSTRAINT fk_recommendations_assistant_message
        FOREIGN KEY (assistant_message_id) REFERENCES chat_messages(id)
);
```

`parsed_constraints` 示例：

```json
{
  "category": "川菜",
  "maxAveragePrice": 100,
  "scene": "朋友聚会",
  "distanceLimit": 5
}
```

建议索引：

```sql
CREATE INDEX idx_recommendations_user_id ON recommendations(user_id);
CREATE INDEX idx_recommendations_session_id ON recommendations(session_id);
CREATE INDEX idx_recommendations_created_at ON recommendations(created_at);
```

## 13. 推荐商家明细表 recommendation_items

一条推荐记录可以包含多个商家。

```sql
CREATE TABLE recommendation_items (

        UNIQUE (recommendation_id, merchant_id),

        UNIQUE (recommendation_id, rank_no)
```

`evidence` 示例：


## 14. 推荐反馈表 recommendation_feedback

保存用户对一次推荐的满意、不满意和文字反馈。

```sql
CREATE TABLE recommendation_feedback (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    recommendation_id BIGINT NOT NULL,
    feedback_type VARCHAR(20) NOT NULL,
    content TEXT,
    reason_category VARCHAR(100),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_recommendation_feedback_user
        FOREIGN KEY (user_id) REFERENCES users(id),

    CONSTRAINT fk_recommendation_feedback_recommendation
        FOREIGN KEY (recommendation_id) REFERENCES recommendations(id),

    CONSTRAINT uk_feedback_user_recommendation
        UNIQUE (user_id, recommendation_id)
);
```

`feedback_type` 可选值：

```text
SATISFIED
DISSATISFIED
```

唯一约束：

```sql
UNIQUE (user_id, recommendation_id)
```

含义：

> 同一个用户对同一次推荐最多保留一条反馈。

用户再次提交时，后端应更新原反馈，不新增第二条记录。

建议索引：

```sql
CREATE INDEX idx_feedback_recommendation_id
    ON recommendation_feedback(recommendation_id);

CREATE INDEX idx_feedback_type
    ON recommendation_feedback(feedback_type);

CREATE INDEX idx_feedback_created_at
    ON recommendation_feedback(created_at);
```

## 15. 反馈重复限制设计

重复反馈采用三层限制：

### 15.1 前端

用户点击提交后，在请求完成前禁用按钮，防止连续点击。

### 15.2 后端

根据：

```text
user_id + recommendation_id
```

查询是否已经存在反馈。

存在时执行更新，不重复插入。

### 15.3 PostgreSQL

使用唯一约束最终兜底：

```sql
CONSTRAINT uk_feedback_user_recommendation
    UNIQUE (user_id, recommendation_id)
```

即使出现并发请求，也不能保存两条重复反馈。

Redis可用于短时间重复请求限制，但不能替代数据库唯一约束。

| 字段 | 类型 |
|---|---|

向量维度根据最终采用的嵌入模型确定。

## 17. Redis Key 初步设计

### 17.1 用户接口限流

```text
rate_limit:user:{userId}:{apiName}:{timeWindow}
```

示例：

```text
rate_limit:user:12:ai_chat:202607140915
```

### 17.2 IP 接口限流

```text
rate_limit:ip:{ip}:{apiName}:{timeWindow}
```

### 17.3 推荐反馈短期重复限制

```text
feedback_lock:{userId}:{recommendationId}
```

### 17.4 热门商家缓存

```text
hot_merchants:{region}:{category}
```

Redis Key 必须设置合理过期时间，避免长期占用内存。

## 18. 初始化脚本目录

PostgreSQL 初始化 SQL 放置于：

```text
scripts/postgres/
```

例如：

```text
scripts/postgres/
├── 01_schema.sql
├── 02_indexes.sql
└── 03_seed_data.sql
```

Docker PostgreSQL 仅在数据卷首次初始化时自动执行该目录中的脚本。

如果数据卷已经存在，修改初始化 SQL 后不会自动重新执行。

## 19. 数据库设计原则

1. PostgreSQL 是正式业务数据的最终来源。
2. OpenSearch 数据可由 PostgreSQL 数据重新构建。
3. Redis 数据允许失效或清空。
4. 重要业务唯一性必须由数据库约束保证。
5. 外键字段必须建立索引。
6. 接口使用小驼峰，数据库使用下划线。
7. JSONB 只用于结构不固定的数据，不应替代所有普通字段。
8. 密码只能保存哈希值，不能明文保存。
9. 删除核心业务数据时应谨慎，后续可加入逻辑删除。
10. 表结构修改后必须同步更新本文档和 SQL 脚本。
