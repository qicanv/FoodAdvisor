# FoodAdvisor 数据库设计

## 1. 文档说明

本文档用于说明 FoodAdvisor 项目的数据存储划分、PostgreSQL 核心表结构、实体关系、唯一约束、索引设计及第一阶段实施范围。

当前版本为 **V0.2（MVP 可开发版）**，在 V0.1 基础上补充了以下内容：

- 商家用户与商家的归属关系
- 多轮对话条件状态与提取历史
- 推荐算法版本、评分明细和来源溯源
- 评论标签及方面级情感关系
- 商家评论摘要及摘要依据
- 评论批量导入任务与失败明细
- AI 模型调用日志
- 更完整的状态、约束、索引和时间字段规范

本版本优先支持以下核心业务闭环：

```text
用户登录
→ 商家与评论数据导入
→ 对话式需求提取
→ 多轮需求补充
→ 商家检索和排序
→ 推荐结果与真实依据展示
→ 用户反馈
→ 评论情感和标签分析
→ 商家评价摘要
→ AI 调用监控
```

## 2. 数据存储划分

### 2.1 PostgreSQL

PostgreSQL 保存正式业务数据，是系统业务数据的最终来源，包括：

- 用户及角色
- 商家用户归属关系
- 商家
- 商家营业时间
- 菜品
- 评论
- 评论分析结果
- 评论标签
- 对话会话
- 对话消息
- 会话条件状态
- 条件提取历史
- 推荐记录
- 推荐商家明细
- 推荐依据
- 推荐反馈
- 商家评价摘要
- 摘要依据
- 批量导入任务
- AI 调用日志

### 2.2 OpenSearch

OpenSearch 保存搜索与 RAG 检索数据，包括：

- 商家搜索文档
- 菜品搜索文档
- 评论搜索文档
- RAG 知识块
- 文本向量
- 检索关键词
- 文档版本与有效状态

OpenSearch 中的数据应能够根据 PostgreSQL 数据重新构建，不能作为正式业务数据的唯一来源。

### 2.3 Redis

Redis 保存临时数据，包括：

- 接口访问次数
- AI 请求限流计数
- 登录失败计数
- 热门商家缓存
- 会话临时状态缓存
- 短时间重复请求标记
- 分布式锁
- 短期结果缓存

Redis 不作为正式业务数据的唯一存储。

## 3. 统一设计规范

### 3.1 命名规范

PostgreSQL 表名和字段名统一使用小写下划线：

```text
users
merchant_id
created_at
recommendation_feedback
```

Java 和前端接口字段使用小驼峰：

```text
merchantId
createdAt
recommendationId
```

### 3.2 主键规范

核心业务表默认使用：

```sql
id BIGSERIAL PRIMARY KEY
```

对外暴露的业务对象应优先使用单独的业务编号，例如：

```text
merchant_code
event_id
trace_id
external_id
```

不能依赖不同环境中可能变化的自增主键作为外部导入唯一标识。

### 3.3 时间字段规范

所有业务时间统一使用：

```sql
TIMESTAMPTZ
```

核心表建议包含：

```sql
created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

`updated_at` 由后端统一维护，执行更新操作时同步修改。

### 3.4 状态字段规范

状态、角色、类型等字段统一使用：

```text
VARCHAR + CHECK 约束
```

不建议第一阶段直接使用 PostgreSQL ENUM，便于后续扩展状态值。

### 3.5 删除策略

核心业务数据原则上使用逻辑停用或归档，不直接物理删除。

可根据需要增加：

```sql
deleted_at TIMESTAMPTZ
```

历史评论、推荐、反馈、分析、日志等数据应保留。

## 4. 核心实体关系

```text
users
 ├── merchant_members ── merchants
 ├── chat_sessions
 │    ├── chat_messages
 │    ├── chat_session_states
 │    └── constraint_extractions
 ├── recommendations
 │    ├── recommendation_items
 │    │    └── recommendation_evidences
 │    └── recommendation_feedback
 ├── import_tasks
 └── ai_call_logs

merchants
 ├── merchant_business_hours
 ├── dishes
 ├── reviews
 │    ├── review_analysis
 │    └── review_tag_relations ── review_tags
 ├── merchant_review_summaries
 │    └── merchant_summary_evidences ── reviews
 └── recommendation_items
```

主要关系：

```text
用户 1:N 对话会话
对话会话 1:N 对话消息
对话会话 1:1 当前会话状态
对话会话 1:N 条件提取历史
用户 N:M 商家（通过 merchant_members）
商家 1:N 营业时间
商家 1:N 菜品
商家 1:N 评论
评论 1:1 当前评论分析结果
评论 N:M 标签
用户 1:N 推荐记录
推荐记录 1:N 推荐商家明细
推荐商家明细 1:N 推荐依据
推荐记录 1:1 当前用户反馈
商家 1:N 评论摘要版本
评论摘要 N:M 原始评论依据
导入任务 1:N 导入行明细
```

# 5. PostgreSQL 表设计

## 5.1 用户表 users

保存普通用户、商家用户、运营人员和管理员。

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100),
    email VARCHAR(255),
    phone VARCHAR(50),
    role VARCHAR(20) NOT NULL DEFAULT 'USER',
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMPTZ,

    CONSTRAINT uk_users_username UNIQUE (username),

    CONSTRAINT ck_users_role
        CHECK (role IN ('USER', 'MERCHANT', 'OPERATOR', 'ADMIN')),

    CONSTRAINT ck_users_status
        CHECK (status IN ('ACTIVE', 'DISABLED', 'LOCKED'))
);
```

说明：

| 字段 | 说明 |
|---|---|
| username | 登录用户名 |
| password_hash | 密码哈希值，禁止存明文 |
| role | USER、MERCHANT、OPERATOR、ADMIN |
| status | ACTIVE、DISABLED、LOCKED |
| deleted_at | 逻辑删除时间 |

建议索引：

```sql
CREATE INDEX idx_users_role_status
ON users(role, status);
```

## 5.2 商家表 merchants

保存商家核心资料。

```sql
CREATE TABLE merchants (
    id BIGSERIAL PRIMARY KEY,
    merchant_code VARCHAR(50) NOT NULL,
    name VARCHAR(200) NOT NULL,
    category VARCHAR(100) NOT NULL,
    cuisine VARCHAR(100),
    rating NUMERIC(3, 2),
    average_price NUMERIC(10, 2),
    review_count INTEGER NOT NULL DEFAULT 0,
    address VARCHAR(500) NOT NULL,
    region_code VARCHAR(50),
    longitude NUMERIC(10, 6),
    latitude NUMERIC(10, 6),
    phone VARCHAR(50),
    description TEXT,
    environment_tags JSONB NOT NULL DEFAULT '[]'::jsonb,
    platform_status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    business_status VARCHAR(30) NOT NULL DEFAULT 'OPEN',
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMPTZ,

    CONSTRAINT uk_merchants_code UNIQUE (merchant_code),

    CONSTRAINT ck_merchants_rating
        CHECK (rating IS NULL OR rating BETWEEN 0 AND 5),

    CONSTRAINT ck_merchants_average_price
        CHECK (average_price IS NULL OR average_price >= 0),

    CONSTRAINT ck_merchants_review_count
        CHECK (review_count >= 0),

    CONSTRAINT ck_merchants_longitude
        CHECK (longitude IS NULL OR longitude BETWEEN -180 AND 180),

    CONSTRAINT ck_merchants_latitude
        CHECK (latitude IS NULL OR latitude BETWEEN -90 AND 90),

    CONSTRAINT ck_merchants_platform_status
        CHECK (platform_status IN ('ACTIVE', 'DISABLED', 'ARCHIVED')),

    CONSTRAINT ck_merchants_business_status
        CHECK (business_status IN (
            'OPEN',
            'CLOSED',
            'TEMPORARILY_CLOSED'
        ))
);
```

说明：

- `platform_status` 表示平台是否允许该商家对外展示。
- `business_status` 表示商家当前经营状态。
- 搜索和推荐时原则上只返回：

```text
platform_status = ACTIVE
business_status = OPEN
```

建议索引：

```sql
CREATE INDEX idx_merchants_name
ON merchants(name);

CREATE INDEX idx_merchants_category
ON merchants(category);

CREATE INDEX idx_merchants_cuisine
ON merchants(cuisine);

CREATE INDEX idx_merchants_region
ON merchants(region_code);

CREATE INDEX idx_merchants_status_category_rating
ON merchants(platform_status, business_status, category, rating DESC);

CREATE INDEX idx_merchants_average_price
ON merchants(average_price);
```

## 5.4 商家营业时间表 merchant_business_hours

保存结构化营业时间，支持“当前是否营业”“深夜营业”等筛选。

```sql
CREATE TABLE merchant_business_hours (
    id BIGSERIAL PRIMARY KEY,
    merchant_id BIGINT NOT NULL,
    day_of_week SMALLINT NOT NULL,
    open_time TIME,
    close_time TIME,
    is_closed BOOLEAN NOT NULL DEFAULT FALSE,
    crosses_midnight BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_business_hours_merchant
        FOREIGN KEY (merchant_id)
        REFERENCES merchants(id)
        ON DELETE CASCADE,

    CONSTRAINT ck_business_hours_day
        CHECK (day_of_week BETWEEN 1 AND 7),

    CONSTRAINT ck_business_hours_value
        CHECK (
            is_closed = TRUE
            OR (open_time IS NOT NULL AND close_time IS NOT NULL)
        ),

    CONSTRAINT uk_business_hours
        UNIQUE (merchant_id, day_of_week, open_time)
);
```

说明：

- `day_of_week`：1 表示星期一，7 表示星期日。
- 跨午夜营业时设置 `crosses_midnight = TRUE`。

建议索引：

```sql
CREATE INDEX idx_business_hours_merchant_day
ON merchant_business_hours(merchant_id, day_of_week);
```

## 5.6 评论表 reviews

保存系统用户评论和外部导入评论。

```sql
CREATE TABLE reviews (
    id BIGSERIAL PRIMARY KEY,
    merchant_id BIGINT NOT NULL,
    user_id BIGINT,
    import_task_id BIGINT,
    external_id VARCHAR(200),
    source_user_key VARCHAR(200),
    rating NUMERIC(3, 2),
    content TEXT NOT NULL,
    source VARCHAR(50) NOT NULL DEFAULT 'SYSTEM',
    review_time TIMESTAMPTZ,
    status VARCHAR(20) NOT NULL DEFAULT 'PUBLISHED',
    moderation_status VARCHAR(20) NOT NULL DEFAULT 'APPROVED',
    risk_level VARCHAR(20),
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMPTZ,

    CONSTRAINT fk_reviews_merchant
        FOREIGN KEY (merchant_id)
        REFERENCES merchants(id)
        ON DELETE RESTRICT,

    CONSTRAINT fk_reviews_user
        FOREIGN KEY (user_id)
        REFERENCES users(id)
        ON DELETE SET NULL,

    CONSTRAINT ck_reviews_rating
        CHECK (rating IS NULL OR rating BETWEEN 0 AND 5),

    CONSTRAINT ck_reviews_status
        CHECK (status IN ('DRAFT', 'PUBLISHED', 'HIDDEN', 'DELETED')),

    CONSTRAINT ck_reviews_moderation_status
        CHECK (moderation_status IN (
            'PENDING',
            'APPROVED',
            'REJECTED'
        )),

    CONSTRAINT ck_reviews_risk_level
        CHECK (
            risk_level IS NULL
            OR risk_level IN ('LOW', 'MEDIUM', 'HIGH')
        )
);
```

说明：

- 外部导入评论没有系统用户时，`user_id` 可以为空。
- `external_id` 用于识别外部来源中的唯一评论。
- 评论原文保存在 PostgreSQL，并同步至 OpenSearch。
- 被隐藏或删除的评论不得继续用于公开展示，但可保留历史分析数据。

建议索引：

```sql
CREATE INDEX idx_reviews_merchant_time
ON reviews(merchant_id, review_time DESC);

CREATE INDEX idx_reviews_user
ON reviews(user_id);

CREATE INDEX idx_reviews_rating
ON reviews(rating);

CREATE INDEX idx_reviews_status
ON reviews(status, moderation_status);

CREATE UNIQUE INDEX uk_reviews_source_external_id
ON reviews(source, external_id)
WHERE external_id IS NOT NULL;
```

## 5.7 评论分析表 review_analysis

保存每条评论当前有效的整体分析结果。

```sql
CREATE TABLE review_analysis (
    id BIGSERIAL PRIMARY KEY,
    review_id BIGINT NOT NULL,
    sentiment VARCHAR(20) NOT NULL,
    confidence NUMERIC(5, 4),
    low_confidence BOOLEAN NOT NULL DEFAULT FALSE,
    keywords JSONB NOT NULL DEFAULT '[]'::jsonb,
    aspects JSONB NOT NULL DEFAULT '[]'::jsonb,
    negative_reason VARCHAR(100),
    model_name VARCHAR(100),
    model_version VARCHAR(100),
    status VARCHAR(20) NOT NULL DEFAULT 'SUCCESS',
    error_message TEXT,
    analyzed_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT uk_review_analysis_review
        UNIQUE (review_id),

    CONSTRAINT fk_review_analysis_review
        FOREIGN KEY (review_id)
        REFERENCES reviews(id)
        ON DELETE CASCADE,

    CONSTRAINT ck_review_analysis_sentiment
        CHECK (sentiment IN (
            'POSITIVE',
            'NEUTRAL',
            'NEGATIVE',
            'MIXED'
        )),

    CONSTRAINT ck_review_analysis_confidence
        CHECK (confidence IS NULL OR confidence BETWEEN 0 AND 1),

    CONSTRAINT ck_review_analysis_status
        CHECK (status IN ('PENDING', 'SUCCESS', 'FAILED'))
);
```

说明：

- `keywords` 和 `aspects` 保存模型原始结构化输出。
- 标签筛选和统计不直接依赖 JSONB，而使用后续的关系表。

建议索引：

```sql
CREATE INDEX idx_review_analysis_sentiment
ON review_analysis(sentiment);

CREATE INDEX idx_review_analysis_analyzed_at
ON review_analysis(analyzed_at DESC);
```

## 5.11 对话消息表 chat_messages

```sql
CREATE TABLE chat_messages (
    id BIGSERIAL PRIMARY KEY,
    session_id BIGINT NOT NULL,
    role VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    message_type VARCHAR(30) NOT NULL DEFAULT 'TEXT',
    request_id VARCHAR(100),
    metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_chat_messages_session
        FOREIGN KEY (session_id)
        REFERENCES chat_sessions(id)
        ON DELETE CASCADE,

    CONSTRAINT ck_chat_messages_role
        CHECK (role IN ('USER', 'ASSISTANT', 'SYSTEM')),

    CONSTRAINT ck_chat_messages_type
        CHECK (message_type IN (
            'TEXT',
            'QUESTION',
            'RECOMMENDATION',
            'ERROR',
            'SYSTEM_NOTICE'
        ))
);
```

说明：

- `request_id` 用于避免用户重复点击造成重复消息和重复 AI 调用。
- 同一个请求 ID 在同一会话内应唯一。

建议索引：

```sql
CREATE INDEX idx_chat_messages_session_created
ON chat_messages(session_id, created_at, id);

CREATE UNIQUE INDEX uk_chat_messages_session_request
ON chat_messages(session_id, request_id)
WHERE request_id IS NOT NULL;
```

## 5.12 会话状态表 chat_session_states

保存当前会话合并后的消费条件及对话阶段。

```sql
CREATE TABLE chat_session_states (
    id BIGSERIAL PRIMARY KEY,
    session_id BIGINT NOT NULL,
    current_constraints JSONB NOT NULL DEFAULT '{}'::jsonb,
    missing_fields JSONB NOT NULL DEFAULT '[]'::jsonb,
    rejected_fields JSONB NOT NULL DEFAULT '[]'::jsonb,
    pending_confirmation JSONB,
    conversation_stage VARCHAR(30) NOT NULL DEFAULT 'COLLECTING',
    version INTEGER NOT NULL DEFAULT 1,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_chat_session_states_session
        FOREIGN KEY (session_id)
        REFERENCES chat_sessions(id)
        ON DELETE CASCADE,

    CONSTRAINT uk_chat_session_states_session
        UNIQUE (session_id),

    CONSTRAINT ck_chat_session_stage
        CHECK (conversation_stage IN (
            'COLLECTING',
            'CONFIRMING',
            'SEARCHING',
            'RECOMMENDED',
```

# 6. 推荐与反馈一致性规则

## 6.1 推荐商家真实性

所有推荐结果中的：

```text
merchant_id
```

必须来自 PostgreSQL 的 `merchants` 表。

推荐时必须过滤：

```text
platform_status != ACTIVE
business_status != OPEN
deleted_at IS NOT NULL
```

的商家。

## 6.2 推荐依据一致性

保存推荐依据时，后端必须校验：

```text
recommendation_items.merchant_id
=
recommendation_evidences.source_merchant_id
```

如果依据来自评论，还必须校验：

```text
reviews.merchant_id
=
recommendation_evidences.source_merchant_id
```

## 6.3 推荐反馈重复限制

重复反馈采用三层限制：

### 前端

用户点击提交后，在请求完成前禁用按钮。

### 后端

根据 `recommendation_id` 查询已有反馈：

- 已存在：更新原反馈
- 不存在：新增反馈

### PostgreSQL

使用唯一约束最终兜底：

```sql
UNIQUE (recommendation_id)
```

Redis 可以用于短时间重复请求限制，但不能替代数据库唯一约束。

# 9. 索引设计原则

1. 所有高频外键查询字段建立索引。
2. 复合索引顺序应匹配实际查询条件。
3. 避免只建立大量单列索引而忽视常用组合查询。
4. OpenSearch 负责全文和语义检索，PostgreSQL 不重复承担复杂全文搜索。
5. 唯一约束和业务幂等优先由 PostgreSQL 保证。

常见查询与索引对应关系：

```sql
-- 查询某商家的最新评论
CREATE INDEX idx_reviews_merchant_time
ON reviews(merchant_id, review_time DESC);

-- 查询某用户的历史会话
CREATE INDEX idx_chat_sessions_user_created
ON chat_sessions(user_id, created_at DESC);

-- 查询会话消息
CREATE INDEX idx_chat_messages_session_created
ON chat_messages(session_id, created_at, id);

-- 查询用户推荐历史
CREATE INDEX idx_recommendations_user_created
ON recommendations(user_id, created_at DESC);

-- 查询商家有效菜品
CREATE INDEX idx_dishes_merchant_status
ON dishes(merchant_id, status);
```


课程项目也可以简化为：

```text
scripts/postgres/
├── 01_schema.sql
├── 02_indexes.sql

```

注意：

> Docker PostgreSQL 只会在数据卷首次初始化时自动执行初始化 SQL。  
> 如果数据卷已经存在，修改 SQL 文件后不会自动重新执行。

开发阶段可通过以下方式处理：

```text
1. 手动执行新增 SQL
2. 删除本地数据卷后重新初始化
3. 后续引入 Flyway 管理数据库版本
```

# 13. 数据库设计原则

1. PostgreSQL 是正式业务数据的最终来源。
2. OpenSearch 数据必须能够由 PostgreSQL 重建。
3. Redis 数据允许过期、丢失或清空。
4. 重要业务唯一性必须由数据库约束保证。
5. 外键字段必须根据查询情况建立索引。
6. 接口使用小驼峰，数据库使用下划线。
7. JSONB 只用于结构可能变化的数据，不能替代需要关联、统计和筛选的关系表。
8. 密码只能保存哈希值，禁止保存明文。
9. AI 推荐中的商家、价格、评分、距离和依据必须来自数据库或系统计算。
10. 大模型只负责意图理解、文本分析和自然语言生成，不直接决定数据库真实性。
11. 用户、商家、评论、推荐等核心数据优先逻辑停用，不直接物理删除。
12. AI 日志不得记录完整密钥、密码或不必要的敏感原文。
13. 表结构修改后必须同步更新本文档、SQL 脚本和接口文档。
14. 第一阶段优先打通核心业务闭环，不一次性创建所有远期功能表。
